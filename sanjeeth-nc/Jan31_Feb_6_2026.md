## Dummy Project Work (Step by Step)

### Part 1: Statistical Model Implementation

#### Step 1: ARIMA/SARIMA Parameter Optimization
- Implemented **auto-ARIMA (pmdarima)** grid search to identify optimal (p, d, q) × (P, D, Q, m) parameters from historical booking data.  
- Conducted **ADF and KPSS stationarity tests** on raw booking series and applied first-order differencing where required.  
- Validated ACF/PACF plots to confirm parameter selection aligned with autocorrelation structure.  
- Ran walk-forward cross-validation with a 7-day forecast horizon, achieving baseline **MAPE of ~18%** on simulated booking data.

#### Step 2: Model Diagnostics & Residual Analysis
| **Diagnostic Check** | **Method** | **Pass Criteria** | **Result** |
|---|---|---|---|
| Residual Normality | Shapiro-Wilk test | p > 0.05 | Validated after log transformation |
| Autocorrelation in Residuals | Ljung-Box test | p > 0.05 for all lags | Passed at lag 10 and lag 20 |
| Heteroscedasticity | Residual variance plot | Constant variance over time | Minor variance increase in peak season |
| Forecast Bias | Mean residual check | Mean ≈ 0 | Negligible bias (-0.3 bookings/day) |

#### Step 3: Exponential Smoothing Baseline
- Implemented **Holt-Winters triple exponential smoothing** with multiplicative seasonality to capture BimRide's proportional demand spikes.  
- Compared additive vs. multiplicative seasonal modes — multiplicative performed better due to higher absolute variance during peak tourist months.  
- Established Holt-Winters as a **lightweight fallback model** for scenarios where compute resources are constrained or rapid predictions are needed.

### Part 2: Advanced Model Development

#### Step 4: Prophet Configuration & Customization
- Configured **custom seasonalities**: weekly (7-day), monthly (30.5-day), and annual (365.25-day) Fourier-order components.  
- Built a **Barbados holiday dataframe** including Crop Over Festival, Independence Day, Errol Barrow Day, Easter weekend, and cruise ship peak arrival days.  
- Added **regressors** for exogenous variables: daily rainfall probability, Grantley Adams Airport flight count, and hotel occupancy index.  
- Tuned **changepoint_prior_scale** (flexibility of trend) and **seasonality_prior_scale** (strength of seasonal effects) through cross-validation.  
- Achieved **MAPE of ~13%** on 7-day forecasts, a significant improvement over the SARIMA baseline.

#### Step 5: LSTM Training Pipeline
| **Pipeline Stage** | **Implementation Detail** | **Configuration** | **BimRide Context** |
|---|---|---|---|
| Data Windowing | Sliding window generator class | 30-day input → 7-day output | Use past month to predict next week |
| Feature Scaling | MinMaxScaler on all input features | Fit on training set only | Prevent data leakage across splits |
| Network Architecture | 2-layer stacked LSTM + Dense output | 64 units per layer, dropout 0.2 | Balance complexity with data volume |
| Training Config | Adam optimizer, MSE loss | Learning rate 0.001, 100 epochs | Early stopping on validation loss |
| Evaluation | Walk-forward on held-out test set | Expanding window, 4 folds | Temporal integrity in evaluation |

- Implemented **early stopping with patience=10** to prevent overfitting on limited historical data.  
- Tested **multivariate LSTM** inputs incorporating weather, day-of-week encoding, and holiday flags alongside raw booking counts.  
- Compared single-step vs. multi-step output architectures — multi-step (direct 7-day output) showed more stable forecasts than recursive single-step prediction.  
- Achieved **MAPE of ~11%** with multivariate inputs, outperforming both SARIMA and Prophet individually.

#### Step 6: Ensemble Model Assembly
- Built a **weighted averaging ensemble** combining SARIMA (weight 0.2), Prophet (weight 0.35), and LSTM (weight 0.45) based on inverse-MAPE weighting.  
- Implemented a **stacking meta-learner** (Ridge Regression) trained on base model predictions to learn optimal combination dynamically.  
- The stacked ensemble achieved **MAPE of ~9%** on the test set — a 50% improvement over the SARIMA baseline.  
- Generated **80% and 95% prediction intervals** from ensemble variance for risk-aware operational planning.

### Part 3: Deployment & Production Readiness

#### Step 7: Model Serialization & Serving
- Serialized trained models using **joblib** (SARIMA, Prophet) and **TensorFlow SavedModel** (LSTM) for reproducible loading.  
- Designed a **FastAPI prediction endpoint** that accepts a date range and returns ensemble booking forecasts with confidence intervals.  
- Implemented **model versioning** with timestamp-tagged artifacts stored in cloud object storage for rollback capability.  
- Created **input validation schemas** (Pydantic) to ensure API requests contain valid date ranges and optional regressor overrides.

#### Step 8: Monitoring, Drift Detection & Retraining
| **Monitoring Component** | **Implementation** | **Trigger Condition** | **Action** |
|---|---|---|---|
| Forecast Accuracy Tracking | Daily MAPE calculation vs. actuals | MAPE exceeds 20% for 3 consecutive days | Alert operations team |
| Data Drift Detection | PSI (Population Stability Index) on input features | PSI > 0.2 on any feature | Flag for investigation |
| Concept Drift Detection | CUSUM test on prediction residuals | Cumulative shift exceeds threshold | Trigger automatic retraining |
| Automated Retraining | Airflow DAG with weekly schedule | Every Sunday at 02:00 UTC | Retrain all base models on latest data |
| A/B Model Comparison | Shadow deployment of new vs. current model | New model MAPE < current for 7 days | Promote new model to production |

- Integrated monitoring metrics into the existing **Snowflake analytics layer** for dashboard visibility.  
- Designed the retraining pipeline as an **Airflow DAG** that pulls latest booking data, retrains all three base models, updates ensemble weights, and deploys if accuracy improves.  
- Implemented **logging and alerting** via webhook notifications to the operations team when forecast accuracy degrades.

## Challenges Faced
1. **ARIMA Grid Search Compute Cost** – Exhaustive parameter search across seasonal orders was time-intensive; switched to stepwise auto-ARIMA to reduce search space.  
2. **Prophet Regressor Alignment** – Ensuring exogenous regressor data (weather, flights) was available at prediction time, not just historically, required building a future-values pipeline.  
3. **LSTM Overfitting** – Initial LSTM models memorized training patterns with near-zero training loss but poor validation performance; resolved with dropout and early stopping.  
4. **Ensemble Weight Stability** – Optimal model weights shifted depending on forecast horizon (1-day vs. 7-day); implemented horizon-specific weighting schemes.  
5. **API Latency** – LSTM inference time (~200ms) created bottlenecks in the prediction endpoint; resolved by pre-computing daily forecasts and caching results.  
6. **Temporal Data Leakage** – Initial cross-validation used random splits instead of temporal splits, producing artificially inflated accuracy; corrected to walk-forward validation.

## BimRide Application (Ride-Hailing)

### Real-Time Booking Predictions
The deployed FastAPI endpoint delivers **hourly and daily booking forecasts** that feed directly into BimRide's dispatch system. Dispatchers can view predicted demand by zone for the next 7 days, enabling pre-emptive driver positioning at Grantley Adams Airport before flight arrivals and at the Port of Bridgetown ahead of cruise ship dockings.

### Automated Operational Adjustments
The ensemble model's confidence intervals enable **tiered operational responses**. High-confidence surge predictions trigger automatic driver notifications and incentive activation, while uncertain forecasts flag the operations team for manual review — balancing automation with human judgment.

### Continuous Model Improvement
The Airflow-orchestrated retraining pipeline ensures forecasting models **evolve with changing demand patterns**. As BimRide expands to new service zones or experiences shifts in tourism trends, the automated drift detection and retraining cycle prevents model staleness without requiring manual data science intervention.

### Cost-Aware Infrastructure
By pre-computing daily forecasts and caching results rather than running real-time inference, BimRide minimizes **compute costs while maintaining forecast freshness**. The lightweight Holt-Winters fallback model ensures predictions remain available even during infrastructure disruptions or model retraining windows.
