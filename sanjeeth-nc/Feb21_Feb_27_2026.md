## Overview
This week focused on **designing and implementing a Retrieval-Augmented Generation (RAG) system** to power intelligent customer-facing and internal-facing assistants for BimRide.  
The objective was to understand RAG's core architecture — how retrieval of relevant context from a knowledge base enhances LLM responses — and build a working pipeline that ingests BimRide's operational documents, FAQs, route guides, and policy information into a searchable vector store.

RAG addresses a critical limitation of standalone LLMs: they hallucinate when asked domain-specific questions they weren't trained on. By **grounding LLM responses in BimRide's actual documentation**, the system delivers accurate, trustworthy answers about routes, pricing, policies, and Barbados travel information without fine-tuning.

## Dummy Project Work (Step by Step)

### Part 1: RAG Architecture & Knowledge Base Design

#### Step 1: RAG Pipeline Fundamentals
- Studied the **two-phase RAG architecture**: retrieval (find relevant documents) → generation (LLM synthesizes answer using retrieved context).  
- Analyzed the difference between **naive RAG** (basic retrieval + prompt stuffing) and **advanced RAG** (query transformation, re-ranking, hybrid search).  
- Mapped BimRide's knowledge sources: ride policies, pricing rules, route guides, Barbados tourism FAQs, driver onboarding materials, and safety protocols.  
- Evaluated the trade-off between **retrieval precision** (returning only highly relevant chunks) and **recall** (not missing important context) for different query types.  
- Designed the target user experience: customers ask natural-language questions via chat and receive instant, accurate answers sourced from BimRide's knowledge base.

#### Step 2: Embedding Model Selection & Evaluation
| **Model** | **Dimensions** | **Max Tokens** | **Strengths** | **BimRide Fit** |
|---|---|---|---|---|
| OpenAI text-embedding-3-small | 1536 | 8191 | High quality, easy API | Best for production deployment |
| OpenAI text-embedding-3-large | 3072 | 8191 | Highest accuracy | Premium use cases, re-ranking |
| Cohere embed-v3 | 1024 | 512 | Multilingual, compression support | Tourist queries in multiple languages |
| sentence-transformers/all-MiniLM | 384 | 256 | Free, local inference | Development and testing |
| BGE-large-en-v1.5 | 1024 | 512 | Strong open-source benchmark scores | Cost-sensitive self-hosted option |

- Selected **OpenAI text-embedding-3-small** for production (quality-to-cost ratio) and **all-MiniLM** for local development and testing.  
- Ran **embedding similarity benchmarks** on sample BimRide queries to validate that semantically similar questions ("How much is a ride to the airport?" vs. "What's the fare to Grantley Adams?") cluster correctly.  
- Tested **dimensionality reduction** (Matryoshka embeddings) to evaluate if lower-dimension vectors maintain retrieval quality at reduced storage cost.

#### Step 3: Vector Database Selection & Setup
- Evaluated vector database options for BimRide's scale (~10K–50K document chunks initially):

| **Database** | **Hosting** | **Filtering** | **Cost** | **Decision** |
|---|---|---|---|---|
| **Pinecone** | Fully managed | Metadata filtering | Free tier + pay-per-use | Selected — managed, low ops burden |
| **Weaviate** | Self-hosted or cloud | Hybrid search (BM25 + vector) | Open source + cloud pricing | Strong alternative |
| **Chroma** | In-process / self-hosted | Basic metadata filters | Free (open source) | Used for local development |
| **pgvector** | PostgreSQL extension | Full SQL filtering | Free (existing Postgres infra) | Considered for future consolidation |

- Deployed **Chroma** locally for rapid iteration during development.  
- Configured **Pinecone** serverless index for production deployment with namespace separation (customer-facing vs. internal knowledge).  
- Implemented **metadata tagging** on all vectors (source document, category, last updated date) to enable filtered retrieval.

### Part 2: Document Ingestion & Chunking Pipeline

#### Step 4: Document Processing & Chunking Strategy
- Built a **document ingestion pipeline** using LangChain's document loaders for Markdown, PDF, and plain text sources.  
- Implemented and compared chunking strategies:

| **Strategy** | **Chunk Size** | **Overlap** | **Retrieval Quality** | **Selected For** |
|---|---|---|---|---|
| Fixed-size (token-based) | 512 tokens | 50 tokens | Baseline — sometimes splits mid-sentence | Initial testing |
| Recursive character splitting | 800 chars | 100 chars | Better — respects paragraph boundaries | General documents |
| Semantic chunking | Variable | Context-aware | Best — preserves meaning units | Policy and FAQ docs |
| Markdown header splitting | Section-based | None | Excellent for structured docs | Route guides, onboarding docs |

- Selected **recursive character splitting (800 chars, 100 overlap)** as the default strategy with **markdown header splitting** for structured documents.  
- Enriched each chunk with **metadata**: source file, section heading, document category, and creation date for filtered retrieval.  
- Processed BimRide's initial knowledge base: 45 route guides, 120 FAQ entries, 15 policy documents, 30 tourism info pages → ~3,200 chunks total.

#### Step 5: Retrieval Pipeline & Query Processing
- Implemented **semantic search** using cosine similarity between query embeddings and stored document vectors.  
- Built a **hybrid search pipeline** combining vector similarity (semantic) with BM25 keyword matching (lexical) using Reciprocal Rank Fusion (RRF) to merge results.  
- Configured **top-k retrieval** (k=5) with a **similarity threshold** (cosine > 0.75) to filter out low-relevance results.  
- Implemented **query preprocessing**: spelling correction, query expansion (adding synonyms like "taxi" ↔ "ride" ↔ "transfer"), and intent classification.  
- Added a **re-ranking step** using a cross-encoder model (ms-marco-MiniLM) to reorder retrieved chunks by relevance before passing to the LLM.

#### Step 6: LLM Integration & Prompt Engineering
- Designed **system prompts** grounding the LLM as a BimRide customer assistant that answers ONLY from retrieved context.  
- Implemented **citation injection** — each response includes source references (e.g., "Based on BimRide's Airport Transfer Guide...") for transparency and trust.  
- Built **guardrails** preventing the LLM from answering questions outside BimRide's domain (competitor pricing, unrelated travel advice) with polite redirects.  
- Configured **temperature=0.1** for factual consistency and **max_tokens=500** to keep responses concise and actionable.  
- Tested with GPT-4o-mini for cost efficiency and GPT-4o for complex multi-hop queries requiring reasoning across multiple document chunks.

### Part 3: Application Layer & Testing

#### Step 7: Chatbot Interface Development
- Built a **FastAPI backend** with `/chat` and `/search` endpoints — `/chat` returns conversational answers, `/search` returns raw retrieved documents.  
- Implemented **conversation memory** using a sliding window (last 5 turns) to maintain context across multi-turn interactions without exceeding token limits.  
- Created a **React chat widget** component (reusable from the dashboard project) with streaming response display, source document links, and feedback buttons (thumbs up/down).  
- Added **typing indicators** and markdown rendering for a polished conversational UX.

#### Step 8: Evaluation & Quality Testing
- Built a **golden test set** of 50 question-answer pairs covering common customer queries, edge cases, and adversarial prompts.  
- Evaluated retrieval quality using **Hit Rate** (correct doc in top-5) and **MRR (Mean Reciprocal Rank)**.  
- Measured end-to-end answer quality using **LLM-as-judge** evaluation (GPT-4o scoring responses for accuracy, completeness, and groundedness on a 1–5 scale).  

| **Metric** | **Naive RAG** | **Advanced RAG (with re-ranking + hybrid search)** | **Improvement** |
|---|---|---|---|
| Retrieval Hit Rate (top-5) | 72% | 91% | +26% |

## Challenges Faced
1. **Chunking Granularity Trade-off** – Too-small chunks lost context needed for complete answers; too-large chunks diluted relevance scores. Required iterative testing to find the 800-char sweet spot.  
2. **Semantic vs. Lexical Mismatch** – Pure vector search missed queries using exact BimRide terminology (route codes, zone names); hybrid search with BM25 resolved keyword-specific retrieval gaps.  
3. **Hallucination Containment** – Even with retrieved context, the LLM occasionally fabricated pricing or schedule details not present in source documents; strengthened system prompts with explicit "do not invent information" guardrails.  
4. **Embedding Cost Estimation** – Processing the full knowledge base through OpenAI's embedding API required careful batching and cost tracking; implemented batch embedding with progress logging.  
5. **Multi-Turn Context Management** – Long conversations caused token limit issues when conversation history was naively prepended; implemented sliding window with summarization for older turns.  
6. **Stale Knowledge Detection** – Documents updated after initial ingestion (pricing changes, new routes) required a re-ingestion trigger; designed a document hash-based change detection system.

## BimRide Application (Ride-Hailing)

### 24/7 Customer Support Automation
The RAG chatbot handles **common customer queries** — pricing estimates, route availability, booking process, payment methods, and Barbados travel tips — without human agent involvement. This reduces support ticket volume while providing instant responses at any hour, critical for serving tourists across multiple time zones.

### Driver Knowledge Assistant
An internal-facing RAG application gives drivers **instant access to onboarding materials, route guides, and policy documents** through a simple chat interface in the driver app. New drivers can ask questions about pickup procedures at Grantley Adams Airport or hotel transfer protocols instead of searching through training PDFs.

### Trustworthy & Transparent Responses
By grounding every response in **cited source documents** and refusing to answer outside its knowledge base, BimRide's RAG system avoids the hallucination risks that plague generic chatbots. Customers see exactly where information comes from, building confidence in the platform's reliability.
