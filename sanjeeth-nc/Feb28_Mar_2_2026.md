## Dummy Project Work (Step by Step)

### Part 1: Production Deployment Infrastructure

#### Step 1: Containerization & Service Architecture
- Containerized the RAG pipeline into **three Docker services**: API server (FastAPI), embedding worker (batch processing), and the React chat widget (Nginx-served static build).  
- Created a **docker-compose configuration** for local development mirroring production topology.  
- Designed the production deployment on **Azure Container Apps** for serverless scaling — containers spin up on demand and scale to zero during low-traffic periods.  
- Configured **health check endpoints** (`/health`, `/ready`) for container orchestration liveness and readiness probes.  
- Implemented **environment-based configuration** using Pydantic Settings — API keys, vector DB credentials, and model selections configured via environment variables, not hardcoded.

#### Step 2: API Gateway & Rate Limiting
| **Component** | **Implementation** | **Purpose** | **Configuration** |
|---|---|---|---|
| API Gateway | Azure API Management | Centralized routing, auth, throttling | Routes `/chat` and `/search` to RAG service |
| Authentication | JWT bearer tokens | Secure access control | Customer app tokens vs. internal driver tokens |
| Rate Limiting | Token bucket algorithm | Prevent abuse and cost overruns | 30 requests/min per user, 1000/min global |
| CORS Policy | Whitelist origins | Restrict cross-origin access | BimRide web app and driver portal only |
| Request Logging | Structured JSON logs | Audit trail and debugging | Log query, latency, token usage per request |

- Implemented **separate rate limit tiers**: customer-facing chatbot (30 req/min), internal driver assistant (60 req/min), admin/testing (unlimited).  
- Added **request validation middleware** sanitizing user inputs before they reach the LLM — blocking prompt injection attempts, excessively long queries, and encoded payloads.  
- Configured **timeout policies** (10-second max per request) to prevent slow LLM responses from blocking the API.

#### Step 3: Latency Optimization
- Profiled the end-to-end RAG pipeline and identified bottleneck breakdown:

| **Stage** | **Avg Latency (Before)** | **Optimization Applied** | **Avg Latency (After)** |
|---|---|---|---|
| Query Embedding | 120ms | Cache frequent query embeddings (Redis) | 15ms (cache hit) / 120ms (miss) |
| Vector Search | 85ms | Pinecone serverless (already fast) | 80ms |
| Re-ranking | 200ms | Batch cross-encoder scoring, limit to top-10 candidates | 130ms |
| LLM Generation | 1200ms | Switch to GPT-4o-mini for standard queries, stream responses | 800ms (first token: 250ms) |
| **Total** | **~1600ms** | **Combined optimizations** | **~1025ms (non-cached) / ~920ms (cached)** |

- Implemented **response streaming** (Server-Sent Events) so users see the answer being typed rather than waiting for the full response — perceived latency drops dramatically.  
- Added **Redis caching** for embedding lookups on frequently asked questions ("How do I book a ride?", "What's the airport transfer price?") — cache hit rate of ~35% in testing.  
- Configured **LLM routing**: simple factual queries → GPT-4o-mini (fast, cheap), complex multi-hop questions → GPT-4o (slower, more capable).

### Part 2: Knowledge Base Lifecycle Management

#### Step 4: Automated Ingestion Pipeline
- Built an **Airflow DAG** that monitors BimRide's document sources (Markdown files in Git, Google Docs, Notion exports) for changes on a daily schedule.  
- Implemented **document hash comparison** — only re-processes and re-embeds documents whose content has changed since the last ingestion run, saving embedding API costs.  
- Created an **ingestion audit log** in Snowflake tracking: document name, chunk count, embedding model version, ingestion timestamp, and processing status.  
- Designed a **manual re-ingestion trigger** (API endpoint + admin UI button) for urgent knowledge updates (e.g., pricing changes effective immediately).

#### Step 5: Knowledge Versioning & Rollback
| **Capability** | **Implementation** | **Benefit** | **BimRide Use Case** |
|---|---|---|---|
| Namespace Versioning | Pinecone namespaces (v1, v2, v3...) | Side-by-side knowledge versions | Test new route guides before going live |
| Blue-Green Deployment | Swap active namespace pointer | Zero-downtime knowledge updates | Switch to updated pricing without service interruption |
| Rollback | Revert namespace pointer to previous version | Instant recovery from bad ingestion | Undo accidental deletion of FAQ entries |
| Metadata Timestamps | `ingested_at` field on every vector | Track freshness per chunk | Flag answers sourced from stale documents |

- Implemented **blue-green knowledge deployment**: new documents are ingested into a staging namespace, validated with the golden test set, then promoted to production by swapping the active namespace pointer.  
- Added a **freshness indicator** in chat responses — if the source document is older than 30 days, the response includes a disclaimer: "This information was last updated on [date]. Please confirm with our team for the latest details."

#### Step 6: Knowledge Gap Detection
- Built a **query logging pipeline** capturing every user question, retrieved documents, and whether the user clicked "helpful" or "not helpful".  
- Implemented **unanswered query detection** — queries where retrieval similarity scores fall below threshold (cosine < 0.7) are logged as knowledge gaps.  
- Created a **weekly knowledge gap report** surfaced in the Power BI dashboard: top 20 unanswered questions ranked by frequency.  
- Designed a **feedback-to-ingestion workflow**: when the content team writes new documentation to address a gap, it flows through the automated ingestion pipeline and the gap is marked as resolved.

### Part 3: Observability & Continuous Improvement

#### Step 7: Production Monitoring & Alerting
- Implemented **structured logging** with correlation IDs tracking each query through embedding → retrieval → re-ranking → generation stages.  
- Built a **RAG-specific monitoring dashboard** (in the React analytics app) displaying:
  - Query volume (hourly, daily)
  - Average response latency (p50, p95, p99)
  - Retrieval hit rate (% of queries with similarity > threshold)
  - User satisfaction rate (thumbs up / total feedback)
  - Token usage and estimated LLM cost per day
- Configured **PagerDuty-style alerts** for: retrieval hit rate dropping below 80%, p95 latency exceeding 3 seconds, error rate above 5%, and daily LLM cost exceeding budget threshold.

#### Step 8: A/B Testing & Iteration Framework
| **Experiment** | **Variant A** | **Variant B** | **Metric** | **Outcome** |
|---|---|---|---|---|
| Chunk size | 800 chars | 1200 chars | Retrieval hit rate | 800 chars won (+4% hit rate) |
| Re-ranker | Cross-encoder (MiniLM) | No re-ranking | Answer accuracy (LLM judge) | Re-ranker won (+0.6 points) |
| Top-k retrieval | k=3 | k=5 | Answer completeness | k=5 won (more thorough answers) |
| LLM model | GPT-4o-mini | GPT-4o | User satisfaction | GPT-4o-mini comparable (4.1 vs 4.3), 5x cheaper |
| Hybrid search weight | 70% vector / 30% BM25 | 50% / 50% | Overall retrieval MRR | 70/30 won (+0.05 MRR) |

- Built an **experiment framework** using feature flags — traffic is split between variants and results are logged with variant IDs for statistical comparison.  
- Established a **weekly RAG review cadence**: review knowledge gap report, evaluate experiment results, plan next iteration — ensuring the system improves continuously rather than stagnating after initial deployment.  
- Documented all configuration decisions and experiment outcomes in a **RAG runbook** for the engineering team.

## Challenges Faced
1. **Cold Start Latency** – Serverless containers on Azure Container Apps took 3–5 seconds to cold start; mitigated with minimum instance count of 1 during business hours and pre-warming scheduled requests.  
2. **Prompt Injection Attempts** – Testing revealed that adversarial inputs ("Ignore previous instructions and...") could bypass guardrails; strengthened input sanitization and added a classifier layer to detect injection patterns.  
3. **Knowledge Staleness Detection** – Determining when a document is "stale" required business-context rules (pricing docs stale after 7 days, route guides after 90 days) rather than a single universal threshold.  
4. **Feedback Signal Sparsity** – Only ~8% of users clicked the thumbs up/down feedback buttons; experimented with auto-prompting feedback after 3 interactions to increase signal volume.  
5. **Cost Visibility** – Tracking per-query LLM costs across GPT-4o-mini and GPT-4o routing required custom token counting middleware; built a cost attribution layer logging estimated cost per request.
